# 智能反馈聚合系统架构书（LLM 理解版）

> 本文档面向 LLM 编程助手，旨在帮助你理解系统的业务逻辑、设计哲学与模块职责，从而生成符合预期的代码。文档聚焦于“做什么”和“为什么”，不涉及具体数据库表结构。  
> **版本**：已根据全部工程决策澄清所有冲突点，可直接作为实现依据。

---

## 一、系统定位

这是一套替代 GitHub Issues 的反馈聚合系统，专门解决以下痛点：

| 痛点 | 本系统的解法 |
|------|-------------|
| GitHub 数据出境合规风险 | 全链路由国内服务器部署 |
| 用户重复提交、issue 泛滥 | AI 语义聚类，自动归并 |
| 违规内容管控压力 | 人工审批准入，所有内容先审后发 |
| 用户看不到问题进度 | 公开看板展示权威状态 |
| 恶意内容溯源难 | 手机号实名登录，全链路可追溯 |

## 二、设计哲学（理解系统的钥匙）

### 2.1 盲盒式提交
用户提交反馈后即与自己的反馈失联。用户端不存在“我的反馈”页面或追踪 ID，提交接口只返回“已收到，感谢反馈！”。用户不知道自己的反馈被分到了哪个 issue，也不知道有多少人提了同样的问题。**用户只负责倾诉，系统负责理解。**

### 2.2 内外有别
- **内部（后台）**：管理员可见原始内容（含可能违规的原文和图片）、关联人数（影响面评估）、提交者手机号（可溯源）。
- **外部（公开看板）**：任何登录用户均可查看，但只展示 AI 生成的客观摘要标题 + 状态标签。**不公开**关联人数、不展示提交者身份、不暴露任何原始反馈内容。

### 2.3 草稿即沙盒
所有新问题先进入“草稿池”，在人工批准前持续演化。每当一个草稿累计接收满 5 条新反馈，立即触发 AI 重新读取该草稿下所有原始反馈原文及图片描述文本，优化标题和描述。直到管理员按下“发布”按钮，草稿才永久冻结并公开。

### 2.4 发布即冻结
一旦公开，标题和描述永不改变。后续即使有更多用户反馈挂载到此 issue，内容也保持不变。给用户提供稳定的“唯一真相”，而非动态变化的讨论帖。

## 三、核心流程（LLM 必须理解的业务逻辑）

### 3.1 用户提交一条反馈会发生什么？

```
用户填写文本内容 + 上传图片（可选）
       ↓
   前端校验：图片 ≤ 5MB，且当日已上传数 < 3 张
       ↓
   【立即返回】“已收到，感谢反馈！”
       ↓
   后端异步处理（用户不等待）：
       ↓
   ① 若包含图片 → 调用多模态模型解析图片内容，生成一段详细的文本解释，
      附在原反馈文本末尾（格式：“【图片描述】…描述内容…”）
       ↓
   ② AI 计算语义向量（基于完整文本，含图片描述），检索两个池子：
      - 已公开的 Issue 池
      - 草稿 Issue 池
       ↓
   ③ 分支判断：
      ├─ 相似度 > 85%（匹配成功）
      │     → 匹配优先级：草稿 > 公开（若同时匹配则归入草稿）
      │     → 挂载到该 Issue（仅内部计数 +1）
      │     → 若目标为草稿，计数满 5 条则立即触发 AI 演化
      │     → 流程结束
      │
      └─ 相似度 ≤ 85%（视为新问题）
            → AI 生成摘要标题 + 描述（系统提示词红线：只客观描述问题，不涉及影响面）
            → 创建一个新的【草稿 Issue】
            → 挂载这条原始反馈（含文本和图片说明）
            → 等待人工审批
```

### 3.2 图片上传限制

- **每用户每日 3 张**（基于手机号计数，UTC+8 自然日）。
- **单张 ≤ 5MB**，超出拒绝上传。
- 上传的图片仅用于辅助问题描述，**不直接展示给任何外部用户**。
- 图片经多模态模型解析后生成的文本描述，会合并到反馈原文中参与后续聚类和摘要生成。
- 管理员在后台可查看原始图片（用于复核）。

### 3.3 草稿发布后会怎样？

```
管理员在后台看到草稿列表
   （按关联人数降序排列，影响面大的排最前面）
       ↓
   查看关联的所有原始反馈快照
   （含提交者手机号、原文、图片及图片描述）
       ↓
   点击【批准并公开】→ 草稿变为公开
       ↓
   ⚠️ 标题和描述被【永久锁定】
   （后续 AI 不会再修改它们）
       ↓
   前端公开看板出现这条问题
```

### 3.4 草稿如何演化？

```
草稿 Issue 每接收满 5 条新反馈时立即触发：
       ↓
   AI 重新读取该草稿下【所有】原始反馈的完整文本（含图片描述）
       ↓
   生成优化后的标题和描述
   （例如：“登录异常” → “因证书过期导致 iOS 端登录失败”）
       ↓
   更新草稿内容（仅后台可见，用户看不到）
```

触发机制为**实时计数触发**，不依赖定时任务。

### 3.5 公开后的反馈怎么处理？

```
公开 Issue 收到了新的匹配反馈：
       ↓
   仅内部关联计数 +1
       ↓
   前端展示：标题不变、状态不变
   （除非管理员手动更新状态为“处理中”或“已修复”）
```

## 四、用户与权限（关键理解点）

### 4.1 用户从哪里来？

- 系统**仅**通过公司内部 SSO 登录，无任何手动注册、无邮箱字段。
- 登录主键是**手机号**，后端扮演“协议转换器”：接收 SSO 回调 → 提取手机号和昵称 → 签发 JWT 给前端。
- 用户第一次登录自动创建账号（若手机号在系统中不存在则自动注册），对用户完全透明。

### 4.2 头像呢？

**没有头像。** 前端用昵称首字母生成彩色圆形 SVG 头像替代。

### 4.3 角色层级

- **普通用户**：可以提交反馈（含图片）、查看公开看板（需登录），无任何后台权限，无“我的反馈”入口，无权限查看自己所提交的历史。
- **管理员**：可以查看草稿池、查看原始反馈快照（含手机号和原始图片）、批准/驳回草稿。

## 五、前端（Vue3）该做什么？

### 5.1 用户端（所有页面均需登录访问）

- **登录页**：只有一个按钮“手机号登录”，跳转到 SSO。
- **反馈提交页**：
  - 一个文本输入框（无标题、无分类）。
  - 图片上传区域：提示“每天最多上传3张，单张不超过5MB”。前端实时校验，超出则禁用上传并提示。
  - 提交按钮。
- **公开看板**：展示所有公开 Issue 的列表，每条显示：
  - AI 生成的标题（如“[Bug] 因证书过期导致无法登录”）
  - 状态标签（待受理 / 处理中 / 已修复）
  - 更新时间
  - **不显示**关联人数、提交者、原始内容、图片
- **不存在“我的反馈”页面或相关入口**，提交后用户与反馈永久失联。

### 5.2 管理后台（仅管理员可见）

- **草稿池列表**：所有草稿 Issue，按关联原始反馈数降序排列。
- **草稿详情页**：显示该草稿关联的所有原始反馈快照，每条包含：
  - 提交者手机号
  - 原始文本（含由图片生成的描述段落）
  - 上传的原始图片（可点击查看大图）
- **操作按钮**：批准并公开 / 驳回并删除（驳回将硬删除草稿及其所有关联的原始反馈和图片）。

## 六、后端（Express）的核心职责

### 6.1 认证模块

- 代理 SSO 登录流程，做协议转换。
- 签发和校验 JWT（payload 含手机号、角色）。
- 自动注册首次登录用户。

### 6.2 反馈提交模块

- 接收文本 + 图片，校验：
  - 图片数量：该用户当日已上传数 + 本次上传数 ≤ 3。
  - 图片大小：每张 ≤ 5MB。
- 将图片保存到服务器，记录元数据。
- 调用多模态模型（兼容 OpenAI 格式，支持视觉理解）对每张图片生成一段详细的文本描述，格式为：“【图片描述】…描述内容…”，附加到用户提交的文本末尾，形成完整反馈内容。
- 将完整反馈内容存入“原始反馈表”（含图片引用和生成的描述），状态为 pending。
- 立即返回 200，异步触发 AI 处理队列。

### 6.3 AI 聚类引擎（最核心）

- 接收原始反馈的完整文本（已拼合图片描述）→ Embedding 向量化 → 检索相似 Issue（草稿池+公开池）。
- 匹配成功 → 挂载到现有 Issue（草稿优先），若目标为草稿则判断是否触发演化。
- 匹配失败 → 调用 Chat 模型生成摘要（系统提示词红线：只描述问题事实，不得涉及影响面、人数）→ 创建草稿 Issue。

### 6.4 草稿管理模块

- 管理员查看草稿池（含关联人数）。
- 查看草稿的原始数据快照（含手机号、原始文本、图片描述、原始图片文件）。
- 批准 → 公开锁定。
- 驳回 → **硬删除**草稿及其所有关联的原始反馈记录和图片文件。

### 6.5 公开看板模块

- 只返回 status = public 的 Issue。
- 只返回标题、状态、时间（不返回关联人数、不返回原始内容，不返回任何图片信息）。
- 接口需要登录校验。

### 6.6 演化触发模块

- 在反馈挂载到草稿时实时检查：若该草稿累计新增反馈数达到 5 条，立即触发 AI 重写摘要（基于所有关联原始反馈的完整文本，包括历史的图片描述）。
- 不存在定时扫描演化任务，无突增告警逻辑。

### 6.7 图片资源管理

- 图片存储路径按用户和日期组织，便于后续清理。
- 驳回草稿时，关联图片文件一并删除。
- 每日上传计数基于手机号按 UTC+8 自然日重置。

## 七、AI 兼容层（火山方舟）

所有 AI 调用通过火山方舟 Chat API（`/chat/completions`）和 Embedding API（`/embeddings`）完成。  
Base URL 默认为 `https://ark.cn-beijing.volces.com/api/v3`，鉴权使用 `Authorization: Bearer <ARK_API_KEY>`。

- **Embedding 模型**：语义向量化（火山方舟 `/embeddings` 端点），用于相似度匹配。强制关闭思维链。
- **命名模型**（唯一允许思维链的模块）：
  - 新问题摘要生成（第一个反馈起名）。
  - 草稿演化摘要优化（每累计 5 条反馈重新起名）。
  - 通过 `thinking: { type: "enabled" }` + `reasoning_effort` 控制思维链开关与强度。
  - **强制系统提示词红线**：只客观描述问题本身，禁止输出任何人数、影响面、频率等暗示信息。
- **多模态模型**（Vision Language Model）：用于图片理解，通过火山方舟 Chat API 传入 base64 图片或图片 URL。强制关闭思维链。系统提示词要求其客观描述图片内容，不做推断，不提及任何人物、隐私信息，聚焦于界面异常、错误提示等与问题相关的视觉元素。

## 八、合规与安全设计

1. **原始内容不公开**：含有潜在违规内容的原文及图片只在后台可见，管理员查看属于依法履职。
2. **关联人数不公开**：防止舆情炒作，用户无法知晓自己“第几个”反馈。
3. **手机号实名溯源**：恶意提交可追溯至具体员工。
4. **审核后再公开**：所有新问题必须经过人工批准才能进入公域。
5. **驳回清理**：违规草稿被驳回时，关联的原始反馈和图片一并硬删除，不留痕迹。
6. **内部公开原则**：公开看板也需登录访问，防止外部爬虫和未授权传播。
7. **图片限制**：每日上传限制防止滥用，图片经多模态模型转译为文本后，原始图片不再被非管理员用户接触，降低隐私泄露风险。

## 九、部署与技术栈总结

- 前端：Vue 3 + Vite
- 后端：Node.js + Express
- 数据库：SQLite（轻量，适合中小团队）
- 文件存储：本地磁盘（图片，可按需扩展至对象存储）
- AI：兼容 OpenAI API 格式（Base URL 指向国内模型厂商），需同时提供 Embedding、Chat、多模态接口
- 认证：JWT + SSO 代理
